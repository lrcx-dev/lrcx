# 用 LLM 把歌词转换成 LRCX

本文件给出“如何提需求”，不是某一种模型专属语法。也可以用作`Prompt of Prompt`。你可以把以下内容和规范草案文件作为提示词发给任意能稳定处理长文本的 LLM，加上歌词让其处理需求。如果你只需专注补充注音或翻译不需要这么长的提示词，描述需求让 LLM 为你编写精简的专用提示词。

文件 [prompts.md](./prompts.md) 有开箱即用的提示词，可查看是否有所需要的。

## 1. 先给模型什么材料

最低建议输入：

- 原始 LRC 或纯歌词文本
- 歌曲元数据
- 目标输出语言
- 是否需要翻译轨
- 是否需要注音轨
- 是否需要逐字时间

建议一次性讲清这些规则：

- 输出必须是合法 LRCX
- `HEAD` 必须先写
- 只使用已声明的标签
- 保留原文歌词，不要擅自改写歌词内容
- 不能编造未知元数据
- 不确定时应留空或回退到 `#more:*`

## 2. 推荐转换流程

1. 先让模型生成最小可用 LRCX
2. 再让模型补翻译轨
3. 再让模型补注音轨
4. 最后让模型做一次规范化与自检

这样通常比“一次要求生成最复杂版本”更稳定。

## 3. 质量要求

复制提示词时，建议总是带上这些要求：

- 不要更换时间标签顺序
- 同一时间点的行要装配成同一个歌词组
- `HEAD` 中先声明再使用
- `#voice`、`#trans`、`#phonetic` 使用规范写法
- `brief`、`supermark`、`full` 不能混淆
- 不要生成未声明标签
- 不要删除原有可用信息
- 如果某项无法确定，请保守处理并显式标注假设

## 4. 可直接复制的提示词

## 4.1 纯 LRC -> 核心 LRCX

```text
请把下面的 LRC 转成 LRCX v1 草案格式。

要求：
1. 先写 HEAD，再写 `[---] v1.0`，再写 BODY。
2. 只输出 LRCX 文件内容，不要解释。
3. 保留原始时间标签顺序与原文歌词。
4. 先生成最小可用版本：title、artist、language、by（若已知）和主歌词行即可。
5. 若元数据未知，不要编造；仅在确有必要时使用 #more:*。
6. 不要生成未声明的标签。

元数据：
- title: <填入>
- artist: <填入>
- language: <填入>
- by: <填入或留空>

原始 LRC：
<粘贴原始 LRC>
```

## 4.2 LRC -> 双语 LRCX

```text
请把下面的 LRC 转成带中英双语轨的 LRCX v1 草案格式。

要求：
1. 在 HEAD 中声明 [#trans:zh,en]。
2. BODY 中每个歌词组保留原文主行，并尽量补上 [#zh] 和 [#en]。
3. 如果某一句无法可靠翻译，请保留主行，不要编造附加轨。
4. 不要改动时间标签。
5. 只输出 LRCX 文件内容。

元数据：
- title: <填入>
- artist: <填入>
- language: <填入原文语言>

原始 LRC：
<粘贴原始 LRC>
```

## 4.3 补注音轨

```text
请在下面的 LRCX 基础上补注音轨，不要改动已有 HEAD 和 BODY 结构。

要求：
1. 若 HEAD 尚未声明注音轨，请补充 [#phonetic:...] 声明。
2. 同时给出：
   - 一个 brief 轨
   - 一个 supermark 轨
   - 一个 full 轨（若要求精细控制）
3. brief 适合简洁注音；full 共享 #timing 语法；supermark 需要绑定原歌词片段。
4. 不要篡改主歌词文本。
5. 只输出完整 LRCX 文件内容。

目标注音：
- type 1: <如 romaji>
- type 2: <如 pinyin>
- 是否需要 full: <yes/no>

输入 LRCX：
<粘贴已有 LRCX>
```

## 4.4 规范化已有 LRCX

```text
请把下面的 LRCX 规范化为一致的 v1 草案写法。

要求：
1. 保持歌词语义、时间语义和已有轨道信息不变。
2. 把 HEAD 中的声明整理到前面。
3. 把 BODY 中同一时间点的行整理为稳定顺序：
   主行 -> #timing -> 翻译 -> 注音 -> #back -> 其他
4. 统一使用 [#voice:...]、[#trans:...]、[#phonetic:...] 的声明格式。
5. 仅在明显冲突时做最小修正，并在文件末尾用 [#note] 简短记录修正点。
6. 只输出完整 LRCX 文件内容。

输入：
<粘贴已有 LRCX>
```

## 4.5 最终自检

```text
请检查下面这份 LRCX 是否符合 v1 草案。

检查项：
1. 是否存在 [---] v1.0 且只出现一次
2. HEAD 声明是否先于 BODY 使用
3. 同一时间点是否出现冲突主行
4. #phonetic 声明与实际轨道是否对应
5. 是否出现未声明标签
6. 时间标签精度是否超过 3 位毫秒
7. #ref 是否只引用前文
8. 输出一个“问题清单 + 修正后的完整 LRCX”

输入：
<粘贴已有 LRCX>
```

## 5. 实用建议

- 让模型先做“结构化补全”，再做“语言润色”。
- 时间敏感内容最好分阶段处理，避免一次提示过于拥挤。
- 对于注音和翻译，要求模型“不确定就保守输出”，通常比强行追求全覆盖更好。

## 6. 版权与隐私提醒

- 只把你有权处理的歌词与元数据发送给外部工具或服务。
- 不要把受版权限制的内容随意上传到不可信平台。
- 如果歌曲、翻译或注音来自第三方贡献，请保留来源说明与贡献信息。
